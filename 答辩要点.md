# SmartVision Stock 智能仓储管理系统 —— 毕业设计答辩要点

> 一句话定位：基于 Vue 3 + Spring Boot 的 **3D 可视化智能仓储管理系统**，具备库存全流程管理、实时 3D 仓库场景、并发安全的出入库与库存预占机制。

---

## 一、项目概述

| 维度 | 内容 |
|------|------|
| 系统类型 | B/S 架构的仓储管理系统（WMS），带 3D 可视化 |
| 前端 | Vue 3 + Vite + Pinia + Element Plus + Three.js（3D 仓库场景） |
| 后端 | Spring Boot 2.7 + Spring Security + JWT + JPA/Hibernate + WebSocket |
| 数据库 | MySQL 8（生产）/ H2（测试）|
| 中间件 | Redis（缓存 + JWT 黑名单）、MinIO（对象存储）|
| 部署 | Docker + docker-compose 一键部署（含 nginx 托管前端）|
| 核心端口 | 前端 8081，后端 8080 |

---

## 二、核心功能模块

1. **基础数据管理**：商品、供应商、仓库、库位（含 3D 坐标）。
2. **入库管理**：入库单创建 → 审核 → 上架，生成货物实例（批次）。
3. **出库管理**：出库单创建 → **库存预占** → 审核 → 确认扣减，支持拣货。
4. **库存管理**：实时库存、调拨、移库、盘点、退货。
5. **3D 可视化**：Three.js 渲染仓库/货架/库位，实时反映库存状态。
6. **实时通知**：WebSocket 推送库存变更、到期预警。
7. **系统管理**：用户/角色/权限（RBAC）、操作日志、文件上传。

## 三、技术亮点（答辩重点讲这几条）

### 亮点 1：库存预占（Reserved Stock）机制 —— 解决"超卖"业务痛点
- **问题**：多个出库单同时想出同一批货，若只在最终确认时才扣库存，会出现"账面够、实际不够"的超卖。
- **方案**：出库单审核阶段就**逻辑锁定**库存（写 `stock_reservation` 记录），可用量 = 库存数量 − 活动预占量。预占带 TTL，`@Scheduled` 定时任务自动释放过期预占，避免"占而不用"锁死库存。
- **价值**：把库存冲突从"确认时"提前到"审核时"发现，符合真实电商/仓储业务逻辑。

### 亮点 2：并发安全 —— 悲观锁防止超卖 / 丢失更新
- **问题**：一个出库单可能拆分成多条明细扣减同一货物实例；并发确认多个单据时，`读-改-写` 存在**丢失更新**。
- **方案**：出库确认前，对所有涉及的货物实例 `id` 先 **`distinct + sorted`（固定加锁顺序防死锁）**，再逐个 `findByIdForUpdate`（`SELECT ... FOR UPDATE` 悲观写锁）加锁，预检查与扣减复用同一批已加锁实体。
- **验证**：写了 `@SpringBootTest` 并发集成测试——8 线程并发确认库存 20（各扣 2），断言 `剩余 = 20 − 确认数×2 且永不为负`；10 线程并发预占也断言不超卖。**全量 104 个测试 0 失败**。
- **为什么不用乐观锁**：原来用 `@Version`，但历史数据版本号为 NULL 导致 Hibernate 自增时 NPE；且预占场景冲突频繁，悲观锁更直接可靠，故统一改为悲观锁。

### 亮点 3：安全设计
- **认证**：JWT（HS384 签名），Access Token + Refresh Token 双令牌，前端拦截器自动刷新。
- **登出即失效**：JWT 黑名单，单机用内存实现，多实例用 **Redis 分布式黑名单**。
- **收紧的 CORS/权限**：Spring Security 只对 auth/swagger 等放行；WebSocket 握手做 JWT 校验，且 **CORS 不再用 `*` 通配兜底**（为空则直接拒绝跨域）。
- **密钥管理**：JWT 密钥、数据库密码全部走环境变量占位（`${...}`），代码零硬编码；启动时缺密钥 fail-fast。

### 亮点 4：实时性与工程化
- **WebSocket 实时推送**：库存变更事件驱动推送到 3D 前端；广播移出数据库事务边界，避免长事务。
- **缓存**：Redis `@Cacheable` 缓存商品等热点数据。
- **可部署性**：Docker 一键起全栈；`/instances` 等大列表接口做了**分页**，避免整表加载。
- **可测试性**：H2 内存库 + `test` profile，集成测试不依赖外部 MySQL/Redis/MinIO。

## 六、可能被问到的问题 & 参考回答

**Q1：你的系统怎么防止超卖？**
A：两层——业务层用库存预占提前锁定，数据层用悲观写锁（`SELECT FOR UPDATE`）保证并发扣减的原子性，并用固定加锁顺序避免死锁，还写了并发集成测试验证。

**Q2：为什么用悲观锁而不是乐观锁？**
A：预占/出库确认场景冲突较频繁，乐观锁失败重试成本高；且历史数据 `@Version` 为 NULL 引发过 NPE。综合可靠性和实现复杂度，选悲观锁。

**Q3：Redis 在系统里起什么作用？**
A：两点——一是缓存热点数据（`@Cacheable`）减轻数据库压力；二是 JWT 登出黑名单的分布式共享，保证多实例部署下登出即时生效。没有 Redis 时自动回退内存实现。

**Q4：WebSocket 如何鉴权？**
A：握手阶段通过子协议/Header 传 JWT，服务端 `HandshakeInterceptor` 校验；同时 CORS 白名单不使用 `*`，为空时直接拒绝，防止任意站点连接。

**Q5：3D 是怎么实现的？性能如何保证？**
A：用 Three.js 按库位坐标生成场景对象，库存变化时更新材质/标签。注意了资源 `dispose` 释放，避免频繁切换场景导致的内存泄漏。

**Q6：数据一致性怎么保证？**
A：核心业务方法 `@Transactional`，扣减/预占在同一事务内；WebSocket 广播移出事务边界，避免推送成功但事务回滚的不一致。

**Q7：项目的测试覆盖情况？**
A：后端 104 个测试（单元 + 集成），含并发预占/超卖专项集成测试，全部通过；前端有 API 助手、缓存、路由等单测。

## 八、可改进方向（主动说，显得有反思）

- 多仓库调拨的库位直接关联可进一步完善（目前部分场景走间接关联）。
- 引入数据库迁移工具（Flyway/Liquibase）替代 `ddl-auto:update`，让 schema 变更可追溯。
- 高并发下可评估从悲观锁转向"预占 + 队列/分段库存"以提升吞吐。
- 增加更多端到端（E2E）自动化测试。
