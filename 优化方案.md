## 2. P0 清单（致命 / 高危）

### 后端
1. **WebSocket 广播写在 `@Transactional` 内部**（StockServiceImpl 共 4 处：inbound/outbound/adjust/move）
   - 后果：事务回滚后客户端仍收到"库存已变更"假通知；广播期间一直占用 DB 连接。
   - 修复：`publishEvent(...)` + `@TransactionalEventListener(phase = AFTER_COMMIT)`，把广播移出事务边界。
2. **调拨 `executeTransfer` 跨仓库误扣 + 无事务**（WarehouseServiceImpl:324-402）
   - 扣减用 `findByGoodsIdAndFrozenFalseOrderByInTimeAsc`（全仓库该 SKU），不按 `sourceWarehouseId` 过滤 → 写"从 A 调出"可能扣 B 的库存；且多步 `save` 无 `@Transactional`，中途异常导致部分落库。
   - 修复：先按「源仓库库位集合」过滤实例；整段加 `@Transactional`（扣减行加乐观/悲观锁）。
3. **默认 JWT 密钥写死在 `application.yml`**（app.jwt.secret 带超长默认值）
   - 后果：仓库泄露即可伪造任意 token。
   - 修复：删除 yml 默认值，缺 `JWT_SECRET` 环境变量时**启动即失败**（fail-fast）。

### 前端
4. **Token / refreshToken 明文存 `localStorage`**（index-helpers.js / router / LoginPage）
   - 后果：任何 XSS（如 `HelpCenter` 的 `v-html`、未来注入点）即可窃取长期冒充。
   - 修复：accessToken 存内存（Pinia）；refreshToken 走 `httpOnly + Secure + SameSite` Cookie，绝不进 JS 可读存储。

### 跨端
5. **调拨 / 退货生成的库存实例 `locationId=null`（孤儿库存）**（WarehouseServiceImpl:383、ReturnServiceImpl:247）
   - 后果：这些货不显示在 3D 库位图、无 `InoutRecord`、无 WS 广播，前后端状态脱节。
   - 修复：调拨/退货时把库存挂到目标仓库空闲 `StorageLocation`（或在 `GoodsInstance` 加 `warehouseId`）；补 InoutRecord + 广播。

## 3. P1 清单

### 后端
- **`getInoutRecords` 按商品名过滤必崩**（StockServiceImpl:443/471）：`root.join("goodsInstanceId",...)` 但字段是普通 `Long`，应改为先查 Goods 再按 `goodsId` 过滤。
- **缺方法级授权 + 对象级越权 (IDOR/BOLA)**：`ReturnController`、`WarehouseController` 的调拨/批准/执行接口未用 `@RequiresPermission`；`getReturnHistory(customerId)`、`NotificationController(userId)` 任意登录用户可读取他人数据 → 从 `SecurityContext` 取当前用户并校验归属。
- **上传无文件类型校验**：`ImageServiceImpl` 直接用原始后缀落盘（对比 `FileUploadController` 有白名单，标准不一致）→ 统一白名单 + `ImageIO` 校验真实类型 + `nosniff`。
- **退货 / 仓库 / 通知无 `@Valid` 输入校验** → 全面加 `@Valid` / DTO 校验。
- **`OrderServiceImpl` 分页 + 关键字搜索逻辑错误**：先按页取再在内存 `contains` 过滤，`totalElements` 不修正 → 改为 DB 层 `LIKE` 分页查询。
- **`WarehouseServiceImpl` 整表加载再 Java 过滤**（inventory/transfer history/delete）→ 改 DB 层带 warehouse 过滤的查询。
- **N+1 普遍**：`StockServiceImpl.convertToDTO`、`OrderServiceImpl` 明细级、`getInoutRecords` 转换层 → `@EntityGraph` / join fetch / DTO 投影 / `@BatchSize`。
- **默认 profile `ddl-auto:update`**（application.yml/dev）→ 主 profile 改 `validate`/`none`，仅本地显式开 `update`。

### 前端
- **`ModelPreview.vue` dispose 不完整**：缺 `forceContextLoss`、几何/材质/纹理未释放、切模型只 remove 不 dispose → 补全。
- **`StockScene.vue` 主渲染循环 & 相机聚焦动画未被 dispose 管控**：裸 `requestAnimationFrame` 在 `onUnmounted` 外继续跑 → 纳入统一追踪/取消。
- **`PrintManager.vue` object URL 永不 `revokeObjectURL`**（750/802/815/656/673）→ 赋值前释放。
- **路由切换无请求取消** → 每个请求绑 `AbortController`，`onUnmounted` abort，避免竞态。
- **`vite.config.js` 无 `manualChunks`**：ElementPlus + three 全打主包 → 拆分 vendor / 按需引入。
- **`allowedHosts` 硬编码 ngrok 子域 + dev 绕过代理**（`.env.development` 用绝对 `:8080/api` 带凭证）→ dev 改相对 `/api` 走代理。

## 4. P2 清单（工程化 / 可维护性）

- **后端无缓存**：`Goods`/`StorageLocation` 高频读适合 `@Cacheable`。
- **`show-sql:true` 默认开启**、缺 Hikari 健康检查配置。
- **业务单号序列器 `static AtomicLong` 重启归零** → 改 DB 序列 / snowflake。
- **异常信息泄露内部细节**（SQL/路径）→ 统一返回，dev/prod 区分。
- **`Map<String,Object>` 返回值反模式 + 重复 `convertToMap`/`errorResult`** → 强类型 DTO + 统一 `ApiResponse` + 自定义 `BizException`。
- **内存态共享状态**（`TokenBlacklist`、`StockWebSocketHandler.sessions`）→ 集群前需 Redis。
- **前端死代码**：`useScene.js`、`VirtualTable.vue`、`utils/debounce.js`、`utils/cache.js`、`errorHandler.handleApiError` 未被引用 → 删除或接回。
- **`HelpCenter.vue:58` `v-html`** 静态安全但属 XSS 反模式 → DOMPurify 或结构化渲染。
- **GET 缓存无限增长 + 失效不彻底**（index-helpers.js）。
- **"任何错误都模拟成功"** 掩盖真实失败（StockScene/PrintManager/Warehouse）。
- **生产残留 `console.*`、魔法数字、巨型组件（Dashboard ~600 行）、无 TS**。
- **缺 ESLint/Prettier CI 门禁、仓库噪声文件**（`nul`、`qc`、`.idea`）。
